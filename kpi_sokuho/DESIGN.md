## Problem Statement

社内コールセンターシステム CTstage 7DX Reporter のリアルタイム指標(着信数・応答中・待ち呼など)を、現在は担当者がブラウザで毎回ログインしてコピペで取得している。これを自動化し、SQLite に時系列スナップショットとして蓄積したい。蓄積データは後段のダッシュボード/通知/集計で使う前提。

## Current Pain (Status Quo)

- 担当者がブラウザでログオン → テンプレート選択 → 画面を見ながら値をコピペ
- 取得頻度は人手に依存(忘れる/間隔がバラつく)
- 蓄積されないので、過去のトレンドが消える
- 「今どうなってるか」を別の場所(Excel など)から見たい時に手間

## Target System (実調査済み)

- **URL**: `http://sccts7dxsql/ctreport` (社内 SSRS 系ホスト名だが実体は CTstage)
- **製品**: OKI CTstage 7DX Reporter (コールセンターレポーティング)
- **認証**: フォームベース `ログオンID` + `パスワード`、本ユーザーはパスワード空欄でログオン可
- **データ取得経路**: `DownloadCsvFile()` JS関数が、ブラウザ内に保持された `ReportDataList[tabIndex].FileData.Get()` から SJIS エンコードの CSV を生成して Blob ダウンロード
- **重要発見**: サーバー側に CSV エクスポート URL は存在しない。データは別の XHR で JSON 等で取得され、CSV 化はクライアントサイドのみ
- **テンプレート**: プライベートスコープに保存される事前定義レポート(5枠あり、現在は未設定)。「テンプレートの呼び出し → レポート作成」でデータが描画される

## Premises (合意済み)

1. 対象は CTstage のリアルタイム指標、テンプレートは事前保存済み or 別途作成
2. CSV はクライアント側でしか作れない → ブラウザ自動化(Playwright)で実現
3. 認証は ID のみで通過、セッション Cookie で維持(起動時 1 回ログオン)
4. 1〜5 分の常時ポーリング、常駐型プロセス
5. SQLite は時系列スナップショットの append-only ストア

## Approaches Considered

### Approach A: Playwright + JS evaluate (採用) ⭐
ヘッドレス Playwright で1度ログオン後、テンプレを切り替えながら `page.evaluate()` でブラウザ内の `ReportDataList` から直接データを読み出す。CSV ボタンクリック → ファイル保存 → 再読み込みの往復をスキップ。

### Approach B: データ API 逆解析 + httpx 直叩き
ブラウザがバックエンドに投げている XHR を特定して、Python から httpx で直接呼び出す。最も軽量だが、ベンダーパッチで API が変わると壊れる。**将来の最適化候補として保留。**

### Approach C: ボタン物理クリック + CSV ファイル読み込み
UI の挙動どおり。ただしファイル I/O、SJIS デコード、ファイル名衝突、ダウンロード完了待ちなど面倒事が増える。

## Recommended Approach (A の詳細設計)

### 技術スタック
- **言語**: Python 3.11+
- **パッケージ管理**: `uv` (`uv init` + `uv venv` + `uv add ...`)
- **依存**: `playwright`, `python-dotenv`
- **DB**: 標準ライブラリ `sqlite3`
- **スケジューラ**: プロセス内 `asyncio.sleep` ループ(まずはこれで運用、必要なら後で Windows タスクスケジューラに移行)

### 実行フロー
```
1. .env 読み込み (URL, ID, PASSWORD, TEMPLATES, POLL_INTERVAL_SEC)
2. SQLite 初期化 (data/kpi.db、スキーマ未作成なら CREATE TABLE)
3. Playwright Chromium 起動 (headless=True)
4. ログオン (1回)
5. ループ:
   for template in TEMPLATES:
     - テンプレート呼び出しダイアログを開く
     - 対象テンプレを選んで「レポート作成」
     - データロード完了まで wait (タブ/メイン領域に内容が出るまで)
     - page.evaluate で ReportDataList[*].FileData.Get() と Name.Get() を取得
     - SQLite に INSERT
   asyncio.sleep(POLL_INTERVAL_SEC)
6. セッション切れ検知時は自動再ログオン (try/except でログオン画面検出)
```

### SQLite スキーマ (現行)

**snapshots テーブル(生 CSV 保管、append-only)**
```sql
CREATE TABLE snapshots (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  captured_at  TEXT    NOT NULL,    -- ISO 8601 UTC
  template_key TEXT    NOT NULL,    -- .env で指定したテンプレ名 (TEMPLATE_HHD 等)
  report_name  TEXT    NOT NULL,    -- CTstage 内部レポート名 (Name.Get(): 着信分析 等)
  target       TEXT    NOT NULL,    -- Target.Get() (電話番号セット / オペレーター名列挙)
  raw_csv      TEXT    NOT NULL,
  row_count    INTEGER
);
CREATE INDEX idx_snapshots_key_time
  ON snapshots(template_key, captured_at DESC);
CREATE INDEX idx_snapshots_report_time
  ON snapshots(report_name, captured_at DESC);
```

**v2: 正規化テーブル(KPI app からの参照用、UPSERT)**

### エラー処理方針
- セッション切れ → 再ログオン1回試行 → ダメなら 1 サイクルスキップ&ログに WARN
- ネットワーク切断 → 1 サイクルスキップ&ログに ERROR、次サイクル続行
- ページ構造変化(セレクタ失効) → ログに CRITICAL + 直近スナップショットを `logs/crash-{ts}.png` に保存 → プロセス終了(タスクスケジューラなら自動再起動)
- SQLite 書き込みエラー → リトライ 3 回、ダメなら CRITICAL ログ

## Open Questions (進捗)

1. ✅ **テンプレート**: 6 個 — `Incom_Analysis`(着信分析-内訳あり 単発), `TEMPLATE_HHD/KMN/SS/TVS`(着信分析+グループ分析 2ファイル), `TEMPLATE_OP`(オペレーター分析 35KB 単発)
2. ✅ **データ用途**: 社内Web → SQLite + 正規化テーブル(v2)が必要。Web 側が SQL or API で読む前提でスキーマ設計
3. ✅ **運用機**: 別マシン (Windows Server 2019)。常時稼働なので PC スリープ問題は無し。Windows サービス化(nssm 等)推奨
4. ✅ **同一 ID 並行ログオン**: 可能。担当者がブラウザで使用中でもスクレイパーが同 ID で並行稼働できる
5. ✅ **機微データ**: 無し。SQLite ファイル保管・バックアップは通常運用でOK

## Success Criteria

- [ ] 7日連続で1分間隔ポーリングが停止せずに完走する
- [ ] SQLite に1万件以上のスナップショットが蓄積される
- [ ] 失われたサイクル(欠測)が1日10回以下
- [ ] 蓄積データから「直近1時間の待ち呼推移」を SELECT 1本で出せる
- [ ] 手作業コピペが完全に不要になる

## The Assignment (次の具体アクション)

実装に入る前に **1つだけ確認すること**: 自分のアカウントで「プライベート」テンプレートを1つ作って、欲しい指標(待ち呼数・応答中数など)が実際にCSVで取れるかを手動で確認する。これが取れない / テンプレ作成にIT部門承認が要る、なら設計を見直す必要がある。

所要 5〜10 分。これが OK なら明日にはスクレイパー本体に着手できる。
